# ROS 2 Workspace Instructions

This repository is the practical ROS 2 workspace companion to the course:

```text
Danncode10/ros2-fundamentals-tutorial
```

The course repository contains lessons, lesson plans, installation guides, quizzes, and teaching material.

This repository contains ROS 2 workspace code that should be cloned and tested inside Ubuntu.

## Target Environment

- Ubuntu 24.04 LTS for ROS 2 builds and runtime tests
- ROS 2 Jazzy inside Ubuntu
- Python beginner nodes with `rclpy`
- `colcon` inside Ubuntu

## macOS and Ubuntu Workflow

The developer setup is split across two machines/environments:

- Codex or Claude runs on macOS.
- ROS 2 development builds and runtime checks happen in an Ubuntu VM running in UTM on the same Mac.

The normal workflow is:

1. Vibe code, edit files, and use Git on macOS.
2. Do not install ROS 2, `colcon`, `rclpy`, or ROS dependencies on macOS.
3. Do not run ROS commands on macOS.
4. Build and test ROS 2 code inside Ubuntu only.

When a task requires Ubuntu, ROS 2, or Linux-specific behavior, do not run that command on macOS. Tell the user that it must be run inside the Ubuntu UTM VM, then provide the exact Ubuntu command.

Ubuntu-only tasks include:

- ROS 2 CLI commands
- `colcon` builds or tests
- sourcing ROS 2 setup files
- installing ROS 2 packages with `apt`
- Linux system package checks
- commands that depend on Ubuntu paths, services, or shell environment

Run these commands in Ubuntu UTM, not macOS:

```bash
colcon build
source install/setup.bash
ros2 pkg list
ros2 run <package_name> <node_name>
ros2 topic list
ros2 node list
```

If working from macOS and ROS validation is needed, explain the Ubuntu UTM commands to run instead of attempting to install or run ROS locally. It is acceptable to edit files and run macOS-safe checks from the shared folder on macOS, but any ROS, Ubuntu, or Linux-specific verification must be deferred to Ubuntu.

## Teaching Standard

Code and documentation in this repo should be beginner-friendly:

- Explain commands before asking the learner to run them.
- Explain expected output and success signs.
- Keep examples tiny before adding abstraction.
- Prefer terminal-based ROS 2 checks before graphical tools.
- Add troubleshooting notes when common beginner errors are likely.

## Storage and Tooling Rules

Keep early examples lightweight.

Do not require these tools in beginner packages:

- Gazebo or Ignition
- Navigation2
- MoveIt
- Docker
- YOLO or AI vision packages
- large simulation worlds
- full ROS desktop stacks

These can be future work after ROS 2 fundamentals are comfortable.

## Workspace Rules

Use this layout:

```text
ros2_ws/
├── src/
├── build/
├── install/
└── log/
```

Only `src/` and documentation should be versioned.

Do not commit:

- `build/`
- `install/`
- `log/`

Those folders are generated by:

```bash
colcon build
```
